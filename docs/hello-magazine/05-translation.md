# 05. 多言語・翻訳戦略（CMSスライド由来）＋マガジンの方針

CMSタグ管理スライドに、プラットフォーム全体の翻訳方針が明記されている。
マガジン（本リポジトリ）の多言語方針はこれに整合させる。

## プラットフォーム全体の方針（スライド要約）

- **ベース言語＝日本語**。対応言語：日本語 / 英語 / 中国語 / 韓国語 / マレー語（5言語）。
- **静的コンテンツ**（タイトル・説明文・タブ名・フォーム文言など）
  → **ローンチ前に各言語を人力で登録**（最初から人力翻訳を入れる）。
- **UGC**（口コミ・コメント・Q&A・コミュニティ）
  → 表示言語に対して **「人力訳 ＞ 自動訳キャッシュ ＞ 即時自動翻訳→キャッシュ」** の優先表示。
  → 翻訳種別バッジ（✅監修翻訳 / 🤖自動翻訳）＋「原文を見る」トグル。
- **翻訳タイミング**＝表示時。キャッシュし、人力訳が入っていれば人力優先。
- **AI翻訳は英語をハブ**にする。API は DeepL or Google Cloud Translate。
- UGC のデータモデル（スライド明記）：
  `original_text` / `original_lang` / `translations{ja,en,ms,ko,zh:{text,source:human|mt,updated_at}}`
  / `translation_status{...}`。管理画面に「監修キュー」。

## マガジン（本リポジトリ）の位置づけと方針

- マガジン記事（LIVE / インタビュー / FAQ / ランキング / エージェント比較）は
  **基本的に静的な編集コンテンツ**（猪口さんが作成）。UGC ではない。
- よって方針は「**静的コンテンツ＝人力翻訳**」に該当 → **手動翻訳でOK**。
  （ユーザー方針「WordPress側は手動翻訳、ACFでフィールド」とも整合）

### 実装方式の選択（要決定 → 03-questions Q1）

| 方式 | 概要 | 向き |
|------|------|------|
| **A. Polylang（推奨）** | 言語ごとに記事を分け、各言語を手動作成。ACFは1セットのまま。 | 繰り返し多い構造的コンテンツ・SEO重視に強い |
| B. ACF 言語別フィールド | 1記事に各言語のフィールドを持たせる | フラットで単純な項目向き。繰り返し（目次/Q&A/順位/エージェント約20項目）には不向き |

- マガジンは **繰り返しフィールドが多い** ため、B だと
  フィールドが言語数×で膨張し、テンプレートも複雑化する。
- Polylang は手動翻訳（自動翻訳しない＝方針どおり）で、言語別URL・hreflang・
  言語スイッチャ・言語別メニューを標準提供 → スライドの「SEO対策」要件にも適合。
- UGC 側の「人力>自動」翻訳ロジックは **アプリ層（口コミ/Q&A/コミュニティ）の課題**であり、
  マガジンの編集CPTとは別レイヤー。混同しない。

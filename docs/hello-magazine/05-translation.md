# 05. 多言語・翻訳戦略（CMSスライド由来）＋マガジンの方針

CMSタグ管理スライドに、プラットフォーム全体の翻訳方針が明記されている。
マガジン（本リポジトリ）の多言語方針はこれに整合させる。

## プラットフォーム全体の方針（スライド要約）

- **ベース言語＝日本語**。対応言語：日本語 / 英語 / 中国語 / 韓国語 / マレー語（5言語）。
- **静的コンテンツ**（タイトル・説明文・タブ名・フォーム文言など）
  → **ローンチ前に各言語を人力で登録**（最初から人力翻訳を入れる）。
- **UGC**（口コミ・コメント・Q&A・コミュニティ）
  → 表示言語に対して **「人力訳 ＞ 自動訳キャッシュ ＞ 即時自動翻訳→キャッシュ」** の優先表示。
  → 翻訳種別バッジ（✅監修翻訳 / 🤖自動翻訳）＋「原文を見る」トグル。
- **翻訳タイミング**＝表示時。キャッシュし、人力訳が入っていれば人力優先。
- **AI翻訳は英語をハブ**にする。API は DeepL or Google Cloud Translate。
- UGC のデータモデル（スライド明記）：
  `original_text` / `original_lang` / `translations{ja,en,ms,ko,zh:{text,source:human|mt,updated_at}}`
  / `translation_status{...}`。管理画面に「監修キュー」。

## マガジン（本リポジトリ）の位置づけと方針

- マガジン記事（LIVE / インタビュー / FAQ / ランキング / エージェント比較）は
  **基本的に静的な編集コンテンツ**（猪口さんが作成）。UGC ではない。
- よって方針は「**静的コンテンツ＝人力翻訳**」に該当 → **手動翻訳でOK**。
  （ユーザー方針「WordPress側は手動翻訳、ACFでフィールド」とも整合）

### 実装方式：Polylang（手動翻訳）で確定 ✅

繰り返しフィールドが多いマガジンでは、ACF言語別フィールド（言語数×でフィールド膨張）より
**Polylang（言語別に記事を分ける）** が保守的。手動翻訳・言語別URL・hreflang・
言語スイッチャを標準提供し、スライドの「SEO対策」要件にも適合。

#### 導入済みの設定（ローカル）
- **Polylang 3.8.5** を導入・有効化（無料版／第三者プラグインのため .gitignore 除外）。
- 言語（DB設定・ベース＝日本語）:

  | slug | 名称 | locale |
  |------|------|--------|
  | `ja` | 日本語（既定） | ja |
  | `en` | English | en_US |
  | `zh` | 中文（简体） | zh_CN |
  | `ko` | 한국어 | ko_KR |
  | `ms` | Bahasa Melayu | ms_MY |

- **翻訳対象のCPT/タクソノミーをコード化**（`inc/polylang.php`の
  `pll_get_post_types` / `pll_get_taxonomies` フィルタ）。管理画面の手動設定に依存せず再現可能。
  - 対象CPT: hello_live / hello_interview / hello_faq / hello_ranking / hello_agent
  - 対象タクソノミー: hello_tag / hello_persona / hello_ranking_type / hello_curriculum / hello_region / hello_price
- 既存投稿・初期タームには既定言語(ja)を割当済み。

#### 別環境（ステージング/本番）での言語追加手順
管理画面 **言語 > 言語** で上表の5言語を追加（ベース=日本語）。
または WP-CLI で `PLL()->model->add_language([...])` を流す
（ローカル投入時のスクリプトは本コミットのコマンド履歴参照）。

### UGC 翻訳は別レイヤー
口コミ/Q&A/コミュニティの「人力>自動翻訳（英語ハブ・DeepL/Google・監修フロー）」は
**アプリ層の課題**で、マガジンの編集CPT（静的・手動翻訳）とは別。混同しない。
